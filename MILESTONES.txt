AISurv Implementation Milestones
================================

Document purpose
----------------
This document tracks implementation progress from the current prototype to a
commercial and enterprise-ready AISurv platform. A milestone is complete only
when all required work and exit criteria are satisfied and verified.

Status values
-------------
[ ] Not started
[-] In progress
[x] Completed
[!] Blocked

M1 - Reliable Camera Management Platform
========================================

Status: [x] Completed
Depends on: M0
Implementation stages: Stages 1 and 2

Objective
---------
Separate the JavaFX client from a modular local edge backend, establish a
versioned local API, and deliver a complete, reliable camera discovery,
registration, configuration, and health lifecycle.

Scope decision: M1 supports credential-free cameras. Authenticated camera storage
and authenticated ONVIF device/media interrogation move to M3, where an OS-backed
secret store is required. Dashboard/incident/event persistence remains owned by
M2/M3, and full observability remains owned by M5. Real-vendor camera validation
is an external release gate because no representative hardware is available.

Required work - service and data foundation
-------------------------------------------
[x] Restructure the build into desktop, edge-core, edge-backend, and API-contract modules.
[x] Introduce a separate Spring Boot edge runtime with an independent lifecycle.
[x] Define versioned command/query APIs and SSE contracts for the desktop client.
[x] Generate or share transport DTOs without exposing persistence entities.
[x] Introduce application-service interfaces inside the edge backend.
[x] Implement CameraApplicationService.
[x] Assign durable EventApplicationService implementation to M2 with event persistence.
[x] Assign DashboardService implementation to M2 with durable event aggregates.
[x] Assign IncidentReviewService implementation to M2/M3 with incidents and evidence.
[x] Keep M1 system health coarse and assign full observability to M5.
[x] Introduce M1 camera, edge-health, runtime-health, and update DTOs/view models.
[x] Remove direct Hibernate and repository access from JavaFX controllers.
[x] Move database bootstrap, repositories, stream lifecycle, and AI ownership out
    of the JavaFX process and into the edge backend.
[x] Keep monitoring active when the desktop disconnects, closes, or restarts.
[x] Bind the edge API to loopback; defer authenticated/TLS LAN operation to M3.
[x] Add edge discovery/configuration, health checks, reconnect behavior, and clear
    desktop states for an unavailable or incompatible edge API.
[x] Add API contract and separate-process integration tests.
[x] Use one edge-owned Hibernate SessionFactory and connection pool.
[x] Add M1 Flyway migrations for camera lifecycle, runtime health, and configuration audits;
    assign zones/events/incidents/settings schemas to M2 and security audit to M3.
[x] Add camera repository transactions, required indexes, and optimistic locking;
    defer pagination until M2 introduces historical query surfaces.
[x] Defer Caffeine until a measured high-read configuration use case exists.
[x] Keep PostgreSQL authoritative and defer optional Redis health projection until needed.
[x] Keep typed worker events and assign the application-wide durable event bus to M2.

Required work - camera lifecycle
--------------------------------
[x] List registered cameras from PostgreSQL in Camera Management.
[x] Support camera registration, editing, enabling, disabling, and deletion.
[x] Detect duplicate cameras and stream URLs.
[x] Validate RTSP URI syntax and bounded protocol connectivity before approval.
[x] Keep credential-free WS-Discovery in M1; assign authenticated ONVIF information/profile queries to M3.
[x] Keep manual validated RTSP approval in M1; assign authenticated ONVIF URI resolution to M3.
[x] Reject authenticated cameras in M1; assign secure credential entry/storage to M3.
[x] Reject credential-bearing endpoints and omit credentials from UI views, errors, and logs.
[x] Start newly enabled cameras without restarting the application.
[x] Stop disabled or deleted cameras immediately.
[x] Persist health state and last-seen timestamps.
[x] Implement online, offline, impaired, recovered, and frozen-stream events.
[x] Track camera uptime and controlled reconnect attempts.
[x] Audit all camera configuration changes transactionally.

Exit criteria
-------------
[x] The JavaFX client and edge backend run as separate JVM processes.
[x] JavaFX UI code does not directly access Hibernate, repositories, Redis, RTSP,
    ONNX Runtime, or evidence storage.
[x] The desktop uses only the versioned edge API for operational commands, queries,
    health, and live updates.
[x] Closing or restarting the desktop does not interrupt camera monitoring.
[x] API compatibility and separate-process failure behavior are covered by tests.
[x] Credential-free cameras can be discovered, approved, validated, saved, edited, and removed.
[x] Camera workers start and stop without an application restart.
[x] Invalid and duplicate camera registrations are rejected clearly.
[x] Camera status reflects actual stream-worker state.
[x] Frozen and disconnected streams generate structured health events.
[x] Credential-bearing endpoints are rejected and credentials do not appear in normal UI output or logs.
[x] V3 health/audit durability is covered by PostgreSQL 16 Testcontainers;
    Redis is not authoritative and is not required for recovery.

Completion record
-----------------
Completed on: 2026-07-19
Verified by: Full Maven verify under JDK 21 with PostgreSQL 16 Testcontainers
Evidence/notes: M1 started on 2026-07-19. Verified slices provide a
four-module reactor, versioned health and camera-query contracts, a loopback-only
Spring Boot edge process, desktop compatibility/unavailable states, and edge-backed
camera inventory, discovery, registration, edit, enable, disable, and delete
commands. Registry mutations use optimistic versions and database uniqueness.
Under JDK 21, 123 unit/controller
test cases execute. With AISURV_RUN_CONTAINER_TESTS=true, the PostgreSQL 16
Testcontainers migration/persistence test and packaged separate-process test pass.
Discovery is single-flight
and command failures use structured API errors. The edge now owns stream and AI
workers and reconciles registry mutations dynamically. Reconciliation preserves
active workers when registry reads fail, cleans up partial startup, and retries
initial startup. API v1.3 publishes reset-first camera-update SSE through a
bounded broadcaster. Edge-mode desktops refresh inventory after updates and
reconnect with bounded backoff while snapshot requests pause during outages.
The desktop has no edge-core dependency or local monitoring fallback. Runtime
health, last-seen, uptime, reconnect attempts, frozen/offline/recovered transitions,
and camera-configuration audits are persisted by V3. Credential-bearing cameras
are rejected until M3. Flyway applies V1 through V3 on clean PostgreSQL 16,
Hibernate validates the resulting schema, and health/audit persistence assertions pass.


M2 - Durable Event-Driven Surveillance Core
===========================================

Status: [ ] Not started
Depends on: M1
Implementation stages: Stages 3 and 4

Objective
---------
Separate capture from inference, support controlled multi-camera processing,
and persist structured events, zones, rules, incidents, and acknowledgements.

Required work - processing pipeline
-----------------------------------
[ ] Separate RTSP frame capture from AI inference.
[ ] Implement real bounded asynchronous frame queues.
[ ] Add worker pools, backpressure, and effective frame-drop policies.
[ ] Use camera or FFmpeg presentation timestamps where available.
[ ] Implement resizing and preprocessing.
[ ] Replace frame-count skipping with time-based inference intervals.
[ ] Add critical, high, normal, and low camera-priority scheduling.
[ ] Reuse model sessions safely where practical.
[ ] Make detector thresholds configurable.
[ ] Measure FPS, inference latency, queue depth, and dropped frames.
[ ] Degrade lower-priority processing first under resource pressure.
[ ] Add repeatable multi-camera soak and overload tests.

Required work - events and zones
--------------------------------
[ ] Persist events with camera, zone, type, severity, and confidence.
[ ] Store model version, rule version, and timestamps with each event.
[ ] Persist acknowledgement, escalation, evidence, and notification state.
[ ] Add zones, camera groups, schedules, and restricted-zone policies.
[ ] Move event rules and thresholds into configurable services and storage.
[ ] Implement cooldowns and duplicate-event suppression.
[ ] Group repeated related events.
[ ] Persist events before evidence or notification processing.
[ ] Connect Event Log to paginated database queries.
[ ] Add event filters for time, camera, zone, type, and severity.
[ ] Open the selected Event Log record in Incident Review.
[ ] Implement transactional acknowledgement and escalation commands.
[ ] Replace hardcoded dashboard counts with persisted aggregates.

Exit criteria
-------------
[ ] Slow inference does not block RTSP capture.
[ ] Queue limits and drop policies work under measured load.
[ ] Critical cameras retain processing priority under pressure.
[ ] Events and acknowledgements survive application restart.
[ ] Duplicate activity does not produce uncontrolled event floods.
[ ] Incident Review opens the operator-selected event.
[ ] Dashboard totals agree with persisted event and camera data.

M7 - Enterprise Hybrid and Selectively Distributed Release
==========================================================

Status: [ ] Not started
Depends on: M6 and demonstrated commercial scaling requirements
Implementation stages: Stage 11

Objective
---------
Deploy the optional multi-tenant cloud control plane and extract only the edge
responsibilities that demonstrate independent scaling, failure-isolation,
hardware-placement, release-cadence, or team-ownership needs for commercial
deployments of approximately 50 to 100+ cameras per site.

Required work
-------------
[ ] Record load, failure-isolation, hardware, and ownership evidence before each
    service extraction.
[ ] Extract capture or AI workers behind stable contracts only where measured
    camera or GPU requirements justify separate deployment.
[ ] Extract evidence, notification, or analytics responsibilities only where
    independent scaling or isolation is demonstrated.
[ ] Keep full video frames local to an edge host or explicitly designed local
    transport; do not route raw frame traffic through the cloud control plane.
[ ] Deploy authenticated cloud APIs for tenants, sites, enrollment, licensing,
    fleet health, remote summaries, notifications, and policy distribution.
[ ] Implement a durable edge outbox, idempotent cloud ingestion, retry, replay,
    ordering, and reconciliation semantics.
[ ] Enforce tenant and site isolation, regional data residency, retention, and
    auditable authorization for every cloud record and command.
[ ] Scale versioned command APIs and SSE subscriptions for supported local and
    remote operational clients.
[ ] Introduce Redis Streams, RabbitMQ, or another broker only if required.
[ ] Define message delivery, ordering, retry, and dead-letter semantics.
[ ] Add independent worker scaling and GPU worker pools.
[ ] Preserve site-local operational PostgreSQL and evidence authority while
    provisioning separate cloud PostgreSQL and approved object storage.
[ ] Centralize fleet-level logs, metrics, and health without making local
    surveillance dependent on cloud availability.
[ ] Package services with containers and Helm charts.
[ ] Add Kubernetes readiness and liveness probes.
[ ] Add horizontal autoscaling.
[ ] Add certificate and secret management.
[ ] Add centralized monitoring, alerting, logs, and traces.
[ ] Implement enterprise backup and disaster recovery.
[ ] Test service failure, network partition, retry, and recovery behavior.

Exit criteria
-------------
[ ] The cloud control plane can be unavailable without stopping local camera
    monitoring, detection, incident handling, or evidence preservation.
[ ] Every extracted service has documented evidence for its independent boundary.
[ ] Selected processing workers scale independently by measured demand.
[ ] Desktop and edge binaries contain no cloud database or service-role secrets.
[ ] Tenant isolation and edge-cloud synchronization recovery are verified.
[ ] Message delivery and failure-recovery semantics are verified.
[ ] GPU and CPU worker pools can be scheduled independently.
[ ] Kubernetes deployments use managed secrets and TLS.
[ ] Central health, logs, metrics, and traces cover all services.
[ ] Enterprise availability, recovery, and camera-capacity targets are met.

Progress update procedure
=========================

1. Change a work item from [ ] to [-] when implementation starts.
2. Change it to [x] only after implementation and verification are complete.
3. Mark blocked work as [!] and describe the blocker in Evidence/notes.
4. Run the milestone's required tests before marking exit criteria complete.
5. Record completion date, verifier, release, and supporting evidence.
6. Mark the milestone complete in Overall progress only when every exit
   criterion is [x].
7. Do not begin dependency-sensitive work from a later milestone unless its
   prerequisite interfaces and data contracts have been completed.
